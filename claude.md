## 6. 핵심 비즈니스 로직

### 6-1. 이벤트 소싱(Event Sourcing) 모델

기준_시간표는 **절대 수정하지 않는다**.  
모든 변경은 **일별_시간표에 행을 추가(append)** 하는 방식으로 기록한다.  
화면에 표시할 때마다 기준_시간표 + 일별_시간표를 합산하여 최종 상태를 계산한다.

```
최종 스케줄 = 기준_시간표 + (일별_시간표 Override 적용)
```

**상태값(status) 종류:**

| 상태 | 의미 |
|------|------|
| `이동(OUT)` | 해당 시간/강좌의 기존 수업을 제거 |
| `이동(IN)` | 해당 시간에 수업을 새로 추가 |
| `대강` | 다른 교사가 원본 교사의 수업을 대신 진행 |
| `보강` | 원본 교사가 자신의 결손 수업을 다른 시간에 직접 진행 |
| `자습` | 자습 감독 지정 |
| `통합` | 두 클래스를 한 교실에서 수업 |
| `취소` | 이전 변경 내역을 원복 (보상 트랜잭션) |
| `정상` | 기준 상태 복구 |

### 6-2. 수업 교체 (True Swap) — 4행 트랜잭션

교체 시 원자적으로 4개의 행이 동시에 삽입됨:

```
사유(txID) = [교체-{타임스탬프}] {입력한 사유}

1. 내 원래 수업 제거:   날짜A, 교시A, 강좌A → 이동(OUT)
2. 내 수업을 타겟 시간에 추가: 날짜B, 교시B, 강좌A → 이동(IN)
3. 타겟 수업 제거:       날짜B, 교시B, 강좌B → 이동(OUT)
4. 타겟 수업을 내 시간에 추가: 날짜A, 교시A, 강좌B → 이동(IN)
```

취소 시 `사유(txID)` 필드로 4개 행을 찾아 모두 `취소`로 상계 처리.

### 6-2-b. 수업 보강 (Real Makeup) — 최대 3행 트랜잭션

원본 교사가 결손 수업을 다른 시간에 직접 진행하는 경우:

```
사유(txID) = [보강-{타임스탬프}] {입력한 사유}

1. 원래 수업 결손 처리: 날짜A, 교시A, 강좌A(원본교사) → 이동(OUT)
2. 타겟 시간에 보강 추가: 날짜B, 교시B, 강좌A(원본교사) → 이동(IN)
3. (선택) 타겟 선생님 수업 제거: 날짜B, 교시B, 타겟강좌(타겟교사) → 이동(OUT)  ← 수업 시간에 진입 시에만
```

**보강 탐색 모드 2가지:**
- **자습 시간:** 해당 학급이 공강이거나 자습으로 설정된 시간 → 타겟 선생님 없음(`비어있음`)
- **수업 시간:** 해당 학급에 다른 선생님 수업이 있는 시간 → 타겟 선생님의 수업을 밀어내고 보강 진입

### 6-3. 강좌코드 형식

```
과목명(학년-반)
예: 영어1(2-3)  →  2학년 3반 영어1 강좌
```

- `extractClassInfo('영어1(2-3)')` → `'2-3'`
- 교체 알고리즘에서 **동일 학급(classInfo)끼리만 교환 가능** 체크에 사용

### 6-4. 교체 후보 탐색 알고리즘 (ScheduleEditor)

```
조건 A: 타겟 교사가 [내 원래 날짜/교시]에 공강인가?
조건 B: 내가 [타겟 교사의 타겟 날짜/교시]에 공강인가?
조건 C: 같은 학급(강좌코드의 괄호 안 학년-반)인가?
→ A && B && C 모두 만족 시 추천 목록에 표시
```

탐색 범위: 당일만 / ±1주일 / ±2주일 (과거 및 주말은 자동 제외)

### 6-5. 보강 후보 탐색 알고리즘 (ScheduleEditor)

```
원본 교사 기준으로 공강인 시간대를 탐색:

[자습 시간 모드]
  → 해당 학급이 공강(수업 없음)이거나,
     dailySchedules에서 '자습' 상태인 시간 필터링

[수업 시간 모드]
  → 해당 학급에 타 교사의 정규 수업이 있는 시간 필터링
  (단, 원본 교사 본인의 수업 시간은 제외)
```

탐색 범위: 당일만 / ±1주일 / ±2주일 (보강은 날짜 범위 설정 가능)

**⚠️ 특수 시간 제외:** 월요일 7교시(`자율(전체)`) / 금요일 7교시(`클럽(전체)`)는 보강 후보에서 항상 제외.  
`baseSchedulesForTargetDay`에서 해당 교시에 특수 강좌코드가 있으면 skip.


## 8. 클라이언트 페이지 구조

### 라우팅 (App.tsx)

```
/           → Login.tsx
/dashboard  → Dashboard.tsx
```

### Login.tsx
- 구글 로그인 탭 / 일반 로그인 탭 UI
- `@react-oauth/google` 기반 실제 Google OAuth 로그인 구현 완료
- 로그인 성공 시 JWT 발급, `localStorage`에 `user` + `token` 저장 후 `/dashboard` 이동
- 학교 방화벽 환경을 위해 일반 로그인 탭 병행 제공

### Dashboard.tsx — 3개 탭

**탭 1: 시간표 조회 (view)**
- 기준일 날짜 선택 → 해당 주(월~금)의 주간 시간표 표시
- 교사 필터: 전체 / 특정 교사 선택
  - 전체 선택 시: 각 셀에 학년별 강좌 수 요약 표시 + 클릭 시 학급 배치 모달
  - 특정 교사 선택 시: 해당 교사의 수업만 표시
- Override(변경) 셀은 초록색 테두리/배경으로 구분 표시

**탭 2: 시간표 수정 (edit)**
- `ScheduleEditor` 컴포넌트를 렌더링
- 탭 상태는 `sessionStorage`에 저장 (새로고침 유지)
- 비학생(`userRole !== '학생'`)만 표시

**탭 3: 시간표 분석 (analyze)**
- `ScheduleAnalysis` 컴포넌트를 렌더링
- 기간 + 과목 선택 → 학급별 실제 수업 시간 집계
- 비학생만 표시

### ScheduleEditor.tsx — 3패널 레이아웃

```
[좌측 패널]          [중앙 액션]         [우측 패널]
Source 선택          수정 타입 선택       Target 추천 목록
- 날짜 선택          - 수업 교체          - 알고리즘 필터 결과
- 교사 선택          - 수업 대강          - 탐색 범위 선택
- 해당일 수업 목록   - 수업 보강          - [선택 적용] 버튼
  (클릭 → 교시 선택)  - 수업 자습
- 변경된 수업 취소버튼 - 클래스 통합
                     - 동교과/타교과 토글 (대강 선택 시만)
                     - 자습시간/수업시간 토글 (보강 선택 시만)
```

**actionType 값 → 동작 매핑:**

| actionType | 한글명 | 탐색 방식 | 날짜 범위 |
|------------|--------|-----------|-----------|
| `exchange` | 수업 교체 | 동일 학급 교사 중 양방향 공강 검색 | 설정 가능 |
| `makeup` | 수업 대강 | 해당 교시에 공강인 타 교사 검색 (동/타교과 필터) | 당일만 |
| `realMakeup` | 수업 보강 | 해당 학급의 자습/공강 또는 수업 시간 검색 | 설정 가능 |
| `selfStudy` | 수업 자습 | 해당 교시에 공강인 타 교사 검색 | 당일만 |
| `merge` | 클래스 통합 | 같은 과목명을 원본 학급에 담당하면서 해당 날짜/교시에 다른 학급에 같은 과목 수업 중인 교사 검색 | **당일만 고정** |

**sessionStorage 저장 항목:**
- `editor_selectedDate`: 마지막 선택 날짜
- `editor_sourceTeacher`: 마지막 선택 교사
- `dashboard_activeTab`: 마지막 활성 탭

## 18. 시뮬레이션 팝업 모달 구조 (2026-04-26 수정)

### 수정 전 문제
- 모달 전체 컨테이너에 `overflowY: 'auto'`가 적용되어, 콘텐츠가 길면 `autoFocus` 스크롤에 의해 헤더(제목 + 취소 버튼)가 스크롤 위로 밀려 사라지는 버그
- 수업 자습 선택 시 모달이 화면 최상단에 붙고 취소 버튼이 보이지 않는 버그

### 수정 후 구조

```
<overlay div>  ← position:fixed, alignItems:'flex-start', padding:40px
  <modal div>  ← height:calc(100vh-80px), overflow:hidden, flex column
    ┌─────────────────────────────────────┐  ← flexShrink:0 (항상 고정)
    │ 제목 + 취소 버튼                     │
    │ 설명 텍스트                          │
    ├─────────────────────────────────────┤  ← flex:1, overflowY:auto (스크롤)
    │ 원본 교사 시간표 │ 대상 교사 시간표  │
    │ (좌우 스크롤: overflowX:auto)        │
    ├─────────────────────────────────────┤  ← flexShrink:0 (항상 고정)
    │ 사유 입력 + 확정 버튼                │
    └─────────────────────────────────────┘
```

**핵심 변경 사항:**
- `height: 'calc(100vh - 80px)'` — 확정 높이로 flex:1 레이아웃이 항상 정상 동작
- `alignItems: 'flex-start'` — 모달이 모든 경우에 동일한 위치(상단 40px)에서 시작
- 헤더·푸터는 `flexShrink: 0`으로 고정, 시간표 테이블만 스크롤

## 19. 클래스 통합 알고리즘 (2026-04-27 재설계)

### 합반 후보 탐색 조건 (3단계)

```
1. 대상 교사가 [selectedDate, selectedPeriod]에 수업 중이어야 함
2. 대상 교사가 sourceClassInfo(예: 1-1) 학급을 담당하는 과목이 있어야 함
3. 그 1-1 담당 과목명이 해당 교시에 가르치는 과목명과 일치해야 함
   (단, 해당 교시 학급이 sourceClassInfo와 달라야 함 — 다른 학급에서 같은 과목 수업 중)
```

**예시:**
- 원본: 김도희 → 1-1 국2 (4월 27일 6교시)
- 강민주: 1-1 영어 담당 + 4월 27일 6교시 1-2 영어 수업 중 → **영어 과목명 일치 → 합반 가능** ✅
- 한승연: 1-1 국1 담당 + 4월 27일 6교시 3-7 화2 수업 중 → **과목명 불일치 → 제외** ❌

### 통합 시뮬레이션 표시 방식
- **원본 교사:** 해당 교시 수업 → 빨간색 취소 표시 (수업 불가)
- **대상 교사:** 기존 수업 항목의 상태를 `통합`으로 변경하여 초록색 강조 (자신의 과목 유지)
- ❌ 원본 교사의 강좌코드를 대상 교사 시간표에 추가하지 않음

### 동교과/타교과 토글 제거
- 통합은 과목명 일치 여부만으로 판단 → 교과 구분 불필요
- 동교과/타교과 토글은 `대강(makeup)` 전용으로만 남김

### 탐색 범위
- **당일만 고정** (다른 날짜의 수업과 통합은 불가능)

## 20. 시간표 분석 기능 (ScheduleAnalysis.tsx) — 2026-04-27 추가

### 기능 개요
기간과 과목을 선택하면 **학급별 실제 수업 시간**을 집계해서 보여주는 분석 도구.

### 입력
- 시작일 / 종료일 (date picker)
- 과목명 (courses 데이터에서 특수 과목 제외한 고유 과목명 목록)

### 출력
- 학년별 그룹으로 묶인 카드 형태 결과
- 최고 시간 학급 → 초록 강조, 최저 시간 학급 → 빨간 강조
- 상단에 전체 최솟값/최댓값 요약 표시

### 집계 알고리즘 (이벤트 소싱 적용)

```
For each date in [startDate, endDate] (평일만):
  dayOfWeek = 요일 계산
  dailyForDate = 해당 날짜의 일별_시간표 항목들
  
  // 취소된 트랜잭션 제외
  cancelledTxIDs = dailyForDate.filter(상태='취소').map(사유)
  activeDaily = dailyForDate.filter(상태≠'취소' AND 사유 not in cancelledTxIDs)
  
  For each period 1~7:
    baseCourses = 기준_시간표[dayOfWeek][period]
    
    // 미실시 처리 상태
    removedCodes = activeDaily.filter(교시=period, 상태 in ['이동(OUT)', '자습', '통합'])
    
    activeCourses = baseCourses (removedCodes 제외)
                  + activeDaily.filter(교시=period, 상태 in ['이동(IN)', '보강'])
    
    activeCourses.forEach(code):
      if courses[code].과목명 === selectedSubject:
        classCounts[extractClassInfo(code)]++
```

**상태별 집계 처리:**

| 상태 | 처리 |
|------|------|
| (오버라이드 없음) | ✅ 집계 |
| `대강` | ✅ 집계 (교사만 바뀌고 수업은 진행) |
| `이동(IN)` | ✅ 집계 (새로 추가된 수업) |
| `보강` | ✅ 집계 (결손 보강 수업) |
| `이동(OUT)` | ❌ 미집계 (수업 제거) |
| `자습` | ❌ 미집계 (자습으로 전환) |
| `통합` | ❌ 미집계 (원래 과목 대신 타 과목 수업) |
| `행사` | ❌ 미집계 (행사로 인해 수업 진행 안 함) |
| `취소` | 해당 txID 전체 트랜잭션 무효화 |

## 23. 백엔드 성능 최적화 및 파싱 버그 수정 (2026-04-26 완료)

### 원본 데이터 파싱 오류 수정
- **문제**: 구글 시트 원본(`2026(1)`)에서 한 교사가 한 시간에 여러 학급을 수업할 경우 (예: 한 셀에 `세계\n2-2\n2-3` 입력), 앱스 스크립트가 첫 번째 학급(`2-2`)만 파싱하고 나머지 학급은 무시하는 문제 발견.
- **해결**: `migrateExistingData` 앱스 스크립트를 수정하여 줄바꿈(`\n`)으로 구분된 모든 학급을 반복문을 통해 각각 고유한 강좌 코드(`세계(2-2)`, `세계(2-3)`)로 생성하도록 개선. 누락 없이 모든 다중 학급 데이터가 `강좌_마스터` 및 `기준_시간표`에 반영됨.

### 캐싱(Caching) 도입을 통한 API 과부하 방지
- **문제**: 데이터베이스를 구글 스프레드시트로 사용하고 있어, 다수의 학생과 교사가 동시에 조회할 경우 구글 API Rate Limit(Quota) 초과 에러(`429 Too Many Requests`) 및 지연 발생 위험 높음.
- **해결**: 
  - 백엔드 서버에 `node-cache` 라이브러리 설치.
  - `/api/data` 조회 엔드포인트에 5분(300초) 수명(TTL)의 인메모리 캐싱 적용. 첫 요청 시에만 구글 API를 호출하고 이후 접속자에게는 캐시된 데이터를 즉시 반환하여 API 호출량을 획기적으로 감소.
  - 사용자가 시간표를 변경하거나 회원가입/상태 변경 등의 업데이트(Write) 작업을 수행하면, 즉시 전체 캐시 데이터를 폐기(Invalidation)하여 최신 상태 동기화 보장.

## 24. 플랫폼 통합 로그인 방식 (SSO) 및 구글 시트 기반 회원 관리 (2026-04-30 추가)

### 기능 개요
서대전여고의 모든 웹앱(스마트 시간표, 학생용 AI 등)을 하나의 거대한 플랫폼으로 통합하고, '서대전여고 통합 플랫폼'에서 한 번 로그인하면(SSO, Single Sign-On) 모든 하위 앱을 재로그인 없이 사용할 수 있도록 하는 전략.
또한, 방화벽 문제로 구글 로그인이 불가능한 환경을 대비해 **일반 로그인**을 도입하며, 회원 정보 관리는 기존처럼 **구글 스프레드시트**를 통해 일괄 관리함.

### 구글 시트 마스터 데이터베이스 (회원 관리) 구조
시트의 A열부터 I열까지 통합하여 사용하며, 관리자(교사)가 엑셀 데이터를 복사·붙여넣기하여 손쉽게 관리할 수 있음.

| 열(Column) | 항목명 | 설명 및 사용법 |
|---|---|---|
| **A열** | 구글 계정 | 구글 간편 로그인 시 매칭되는 계정 이메일 |
| **B열** | 일반 ID | 방화벽 환경에서 사용하는 일반 로그인 아이디 (주로 학번 또는 교사 이메일 등) |
| **C열** | 비밀번호 | 일반 로그인용 비밀번호. 초기에는 생년월일이나 학번 등으로 일괄 부여. (서버에서 해싱하여 저장 권장) |
| **D열** | 이름 | 실제 사용자 이름 |
| **E열** | 고유식별자 | 학번(예: 10101) 또는 교사ID(예: T-038). 권한 분기 및 개인별 시간표 연동의 핵심 키 |
| **F열** | 권한 | 관리자 / 업무담당자 / 교사 / 학생 / 학부모 / 졸업생 등 접근 가능한 메뉴를 제어하는 권한 |
| **G열** | 상태 | Active(정상) / Inactive(정지). 전학, 졸업, 퇴사 시 이 값을 변경하여 즉각 접근 차단 |
| **H열** | 가입일시 | 계정이 최초 생성된 날짜 및 시간 (감사 및 모니터링 용도) |
| **I열** | 비번변경필요 | TRUE/FALSE 값. TRUE일 경우 로그인 즉시 강제로 비밀번호 변경 페이지로 리다이렉트 처리 |

### 관리적 장점
- **신속한 권한 통제**: 구글 시트에서 열 한 줄만 지우거나 상태를 '정지'로 바꾸면, 모든 연동 앱에서 즉시 접근이 차단됨.
- **SSO 연동**: 메인 웹앱에서 토큰(Token)을 발급하고 각 서브 앱들이 해당 토큰을 검증하는 방식 적용 가능.

## 25. 향후 개발 계획: 서대전여고 통합 플랫폼(SSO) 분리 연동 (2026-04-30 확정)

### 현재 구조의 한계
현재 스마트 시간표 앱은 서버 내부(`server/index.js`)에서 직접 사용자 구글 시트를 조회하여 로그인을 처리하고 있습니다. 
앞으로 학생용 AI, 급식 관리 등 다수의 앱이 늘어날 경우, 각 앱마다 로그인 기능을 따로 개발하고 유지보수해야 하는 문제가 발생합니다.

### 마이크로서비스 아키텍처(MSA) 및 SSO 도입 계획
시간표 프로젝트 폴더와 완전히 분리된 별도의 **[통합 플랫폼 서버 프로젝트]**를 구축합니다.
- **인증 분리**: 비밀번호 해싱(`bcrypt`), 회원가입, 구글/일반 로그인은 오직 '통합 플랫폼 서버'에서만 전담합니다.
- **슈퍼 앱 연동**: 사용자가 통합 플랫폼에서 로그인을 성공하면 JWT 토큰을 발급받고, 플랫폼 메인 화면에서 '시간표 아이콘'을 누르면 해당 토큰을 들고 스마트 시간표로 리다이렉트됩니다.
- **시간표 앱의 변화**: 스마트 시간표 서버는 더 이상 직접 로그인 검사를 하지 않고, 플랫폼 서버에서 만든 JWT 토큰의 유효성만 검증하여 즉시 접속을 허용합니다. (두 서버 간 `JWT_SECRET` 키를 동일하게 맞춤)
